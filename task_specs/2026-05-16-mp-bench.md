## Functional Spec:
- now that we have the ability to measure things, I need something to measure:
    - generate load against the message proxy service so that we may populate the dashboards and discover the message proxy servers load characteristics
- the measurements should be useful and reflective of user behavior:
    - no coordinated omission
        - requests that take a long time to send or responses that take a long time to process should not hinder the benchmarking tool from sending more requests
    - generated load should sample from an exponential distribution
- goals:
    - configure the number of clients
    - configure the number of documents
    - configure the rate at which clients send operations
    - configure the rate at which clients disconnect and reconnect
    - configure the duration of the test
- Non goals:
    - multiple instances of the message proxy service 
    - state persisted on the client between connections to the message proxy service
        - this means that we will not load test the client sync step two behavior
    - readable / semantically consistent documents
    - loads that dynamically change
    - metrics aggregated on the client side

## Technical Spec:
- write a binary crate that:
    - reads configuration from the terminal
    - creates a number of message proxy service clients based on the provided configuration
    - has a limited set of preset configurations
- make operations and reconnections stochastic instead of fixed timer based
    - we want to test steady state load, not instantaneous bursts of reconnections or operations
    - we will test a thundering heard / reconnection-storm load profile but we will do so explicitly
    - generate events such as "send message" and "disconnect then reconnect" by sampling from an exponential distribution, waiting for the sampled amount of time, then publishing an event
    - events are processed fairly and eagerly by some single threaded worker
- avoid coordinated omission
    - measure response time instead of service time
    - decouple response processing from request generation
    - if the rate at which requests and responses are processed is sufficiently large (larger than the rate at which events are published), then having a single threaded worker processing events may lead to delay between when the event is published and when the event is processed
        - I don't think this problem can be solved because the crdt framework requires serial execution of updates. So we must process received messages and generate sent messages sequentially.
